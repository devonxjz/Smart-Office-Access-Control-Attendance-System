# PRD: Google Sheets Backend Integration

## Problem Statement
The application needs a lightweight, free backend to store and retrieve data for the Smart Office Access Control Attendance System, specifically for attendance ("bảng chấm công") and employee ("nhân viên") records. A traditional database like PostgreSQL is currently too heavy or not desired for this MVP stage. Additionally, the ESP32 hardware needs a secure, low-bandwidth way to verify access without exposing the public GAS URL to manipulation.

## Solution
Integrate Google Sheets as the primary database using Google Apps Script (GAS) as a serverless API layer. 
- The frontend will communicate with the GAS Web App via HTTP, hiding the Google Sheets complexity behind a `GoogleSheetsClient` infrastructure layer and typed Domain Services.
- The ESP32 will use a specific `checkAccess` endpoint with a pre-shared secret token to securely verify UIDs against the employee sheet without downloading the entire list.

## User Stories
1. As an employee, I want my attendance access times recorded in the sheet, so that my work hours are tracked accurately.
2. As a system administrator, I want to view the attendance sheet ("bảng chấm công"), so that I can monitor employee presence and late arrivals.
3. As a system administrator, I want to view and manage the employee list ("nhân viên"), so that the system recognizes valid access attempts.
4. As the frontend application, I want to append new rows to the attendance sheet via API securely, so that data isn't lost or corrupted.
5. As the frontend application, I want to fetch data from specific sheets, so that the UI can display current records.
6. As an ESP32 device, I want to verify an RFID UID against the employee list via a secure endpoint, so that I can quickly grant or deny physical access without downloading the entire list.

## Implementation Decisions

*   **System Architecture**:
    *   **ESP32 Hardware** calls GAS API `checkAccess` endpoint.
    *   **Frontend UI** calls **Domain Services** (`AttendanceService`, `EmployeeService`).
    *   **Domain Services** validate/transform data and call **GoogleSheetsClient**.
    *   **GoogleSheetsClient** handles HTTP fetch to the GAS Web App, including retries, timeouts, and error normalization.
    *   **Google Apps Script API** acts as the backend router, converting JSON requests to sheet row operations and vice versa.

*   **Backend (GAS) Design**:
    *   Endpoints handle requests like `GET ?action=read&sheet=attendance`, `POST ?action=append&sheet=attendance`, and `GET ?action=checkAccess&uid=ABCD&token=SECRET`.
    *   **Security**: All requests from hardware (ESP32) MUST include a pre-shared `token` to prevent unauthorized API calls.
    *   Strict JSON response format for all endpoints: 
        ```json
        { "success": boolean, "data": [], "message": string }
        ```
    *   GAS project structured into separate functional files (`router.gs`, `attendance.gs`, `employee.gs`, `response.gs`, `sheet-utils.gs`).
    *   No business logic in `doGet` or `doPost`. They only route requests.

*   **Frontend Infrastructure (`src/infrastructure/google-sheets.client.ts`)**:
    *   Implements core methods: `read(sheetName)`, `append(sheetName, data)`.

*   **Data Model & Schema**:
    *   Every row in Google Sheets MUST have a unique `id` (e.g., UUID generated by frontend or GAS), `employee_id`, and `timestamps`. 
    *   We do not rely on row indexes for updates/deletes, as rows can be sorted or shifted.

## Testing Decisions
*   **GoogleSheetsClient**: Unit test retry logic, timeout handling, and JSON serialization using a mocked `fetch` adapter.
*   **Domain Services**: Unit test the transformation logic (e.g., mapping raw sheet string `"time_access": "08:19"` to UI domain state `{ "status": "Late", "lateMinutes": 19 }`). Ensure UI model mapping is pure and predictable.
*   Good tests will only test external behavior (the mapped UI objects) and not the internal Google Sheets fetch implementation details.

## Out of Scope
*   OAuth2 user login for Google Sheets (GAS Web App acts on behalf of the developer/service account).
*   Real-time websocket updates (polling or manual refresh will be used instead).
*   Complex relational queries (joins) between sheets.

## Further Notes
*   This architecture explicitly decouples the UI from Google Sheets. If we migrate to Supabase, Firebase, or PostgreSQL later, only the `GoogleSheetsClient` implementation and API credentials need to be swapped out. The Domain Services and UI remain untouched.
